## 실행 모드

### 회고 모드 (Retrospective)

작업을 마친 뒤 전체 대화 기록을 분석하여 스킬을 생성합니다.

```
/skillify [프로세스 설명]
```

**언제 사용하나요?**
- 작업을 끝낸 뒤 "이거 다음에도 쓸 수 있겠다"고 느꼈을 때
- 이미 완료된 작업을 스킬로 저장하고 싶을 때
- 프로세스 설명은 선택사항 — 생략하면 대화에서 자동 추론

**예시:**
```
> (Spring Batch Job을 생성하는 작업을 완료한 상태)
> /skillify Spring Batch Job 생성 자동화
```

### 실시간 기록 모드 (Real-time Recording)

작업 시작 전에 기록을 시작하고, 완료 후 해당 구간의 대화만 분석합니다.

```
/skillify-start        # 기록 시작
... 작업 수행 ...
/skillify [설명]       # 기록 종료 및 스킬 생성
```

**언제 사용하나요?**
- 작업 시작 전에 "이건 스킬로 만들어야겠다"고 미리 알 때
- 긴 대화에서 특정 구간만 캡처하고 싶을 때
- 이전 대화 내용이 스킬과 무관한 잡담이나 다른 작업을 포함할 때

**예시:**
```
> (이전에 다른 작업들을 했음)
> /skillify-start
> [SKILLIFY RECORDING STARTED] 메시지가 표시됨
> API 서버의 health check 엔드포인트를 호출해서 상태를 모니터링해줘
> (... 작업 수행 ...)
> /skillify API 모니터링 자동화
```

### 두 모드의 차이

| 항목 | 회고 모드 | 실시간 기록 모드 |
|------|-----------|-----------------|
| 분석 범위 | 전체 대화 | 마커 이후 대화만 |
| 사전 준비 | 불필요 | `/skillify-start` 호출 |
| 적합한 상황 | 짧은 대화, 단일 작업 | 긴 대화, 복합 작업 |
| 정확도 | 관련 없는 내용 포함 가능 | 해당 작업만 정밀 캡처 |

## Phase 0: 컨텍스트 재구성

`/skillify` 호출 시 자동으로 수행됩니다. 사용자 개입 없이 백그라운드에서 대화를 분석합니다.

### 수집하는 정보

#### 1. 대화 기록 (항상)
- **사용자 의도**: 무엇을 하려고 했는지
- **수정 이벤트**: 방향을 바꾼 지점 ("아니 그거 말고...", "이렇게 바꿔줘")
- **입력 파라미터**: 매번 달라질 수 있는 값 (파일 경로, URL, 설정값)

#### 2. Git artifact (`.git` 존재 시)
- 변경된 파일 목록
- 커밋 메시지
- diff 내용

#### 3. 실행된 도구/명령어
- Bash 명령어 (빌드, 테스트, 배포 등)
- 파일 조작 (Read/Edit/Write)
- 외부 호출 (WebFetch, curl)

### 작업 유형 자동 감지

프로젝트 파일이 아닌 **실제 수행된 작업**을 기반으로 도메인을 판별합니다.

예를 들어:
- Java 프로젝트에서 Python 스크립트를 작성했다면 → **Python** 도메인으로 감지
- git 저장소가 없는 곳에서 curl로 API를 호출했다면 → **API 자동화** 도메인으로 감지
- PDF를 읽고 요약했다면 → **문서 처리** 도메인으로 감지

복합 작업이면 여러 도메인을 동시에 참조합니다.

## Phase 1: 인터뷰 과정

Skillify는 4라운드의 구조화된 인터뷰를 통해 스킬의 세부사항을 확정합니다.

### 라운드 1: 고수준 확인

분석 결과를 바탕으로 제안합니다:
- **스킬 이름** (예: `spring-batch-job-creator`)
- **설명** (예: "Spring Batch Job을 생성하고 스케줄링 설정을 구성")
- **목표** 및 **완료 기준** (예: "Job이 빌드되고 테스트 통과")

사용자는 확인하거나 수정할 수 있습니다.

### 라운드 2: 구조 및 옵션

- **단계 목록**: 식별된 작업 순서를 번호로 제시
- **인자(arguments)**: 매번 달라지는 입력값 제안 (예: `job-name`, `cron-expression`)
- **실행 컨텍스트**: inline (사용자 개입 가능) vs fork (자율 실행) 추천
- **저장 위치**: 프로젝트별 vs 개인 글로벌

### 라운드 3: 스텝별 상세

각 단계에 대해 질문합니다:
- **성공 기준**: 이 단계가 완료되었다는 증거
- **산출물**: 다음 단계가 필요로 하는 데이터/파일
- **사람 확인점**: 되돌릴 수 없는 작업 전 확인 여부
- **병렬 가능**: 독립적인 단계는 동시 실행
- **규칙**: 반드시 지켜야 할 제약 조건

### 라운드 4: 트리거 및 엣지케이스

- **호출 조건**: 언제 이 스킬을 사용해야 하는지
- **트리거 문구**: 자연어 예시 (예: "배치 잡 만들어줘")
- **실패 대응**: 에러 발생 시 행동
- **도메인 확인**: 자동 감지 결과가 맞는지

## Phase 2: 스킬 생성

인터뷰 결과와 도메인 템플릿을 결합하여 SKILL.md를 생성합니다.

### 생성되는 파일 구조

```yaml
---
name: skill-name              # 소문자, 하이픈, 최대 64자
description: >                # 트리거 정보 포함, "Use when..." 패턴
  ...
allowed-tools:                # 최소 권한, 세분화 패턴
  - Bash(./gradlew:*)
  - Read
  - Edit
argument-hint: "<args>"       # 인자 있을 때만
arguments:                    # 인자 있을 때만
  - arg-name
context: fork                 # 자율 실행일 때만 (inline은 생략)
---

# Skill Title
## Inputs
## Goal
## Steps (각 스텝에 Success criteria 필수)
## Rules (교정 이벤트에서 추출)
```

### 도메인 템플릿 활용

감지된 도메인의 템플릿에서 **공통 패턴**을 자동으로 스킬에 반영합니다:
- 빌드/테스트 명령어
- 디렉토리 구조 관례
- 도구 권한 (allowed-tools)
- 일반적인 주의사항

사용자가 대화에서 다르게 작업한 부분은 템플릿보다 우선합니다.

## Phase 3: 리뷰 및 저장

1. 완성된 SKILL.md를 코드블록으로 출력
2. 사용자가 확인하면 지정된 위치에 저장
3. 저장 후 호출 방법 안내: `/<skill-name> [arguments]`

## 실전 예시

### 예시 1: Spring Batch Job 생성 자동화

```
# 1. 작업 수행
> Spring Batch Job을 만들어줘.
> 매일 자정에 users 테이블에서 통계를 집계하는 거야.
> Reader는 JdbcCursorItemReader, Writer는 JdbcBatchItemWriter로.
> (... Claude가 작업 수행 ...)

# 2. 스킬 캡처
> /skillify Spring Batch Job 생성 자동화

# 3. Skillify가 분석하여 인터뷰 진행
# 4. 생성된 스킬: spring-batch-job-creator

# 5. 다음부터는:
> /spring-batch-job-creator user-stats "0 0 * * *"
```

### 예시 2: API 모니터링 자동화

```
# 1. 기록 시작
> /skillify-start

# 2. 작업 수행
> 우리 API 서버 3개의 health check를 호출해서 상태를 확인해줘.
> https://api1.example.com/health, api2, api3
> 응답 시간이 2초 넘으면 경고로 표시하고, JSON으로 결과 저장해줘.
> (... Claude가 작업 수행 ...)

# 3. 스킬 캡처
> /skillify

# 4. 생성된 스킬: api-health-monitor
```

### 예시 3: 문서 정리 자동화

```
# 1. 작업 수행
> 이 PDF 보고서를 읽고 핵심 수치를 추출해서 CSV로 만들어줘.
> (... Claude가 PDF 분석 및 CSV 생성 ...)

# 2. 스킬 캡처
> /skillify 보고서 데이터 추출

# 3. 생성된 스킬: report-data-extractor
```

## 고급 기능

### 복합 도메인 작업

하나의 작업이 여러 도메인에 걸칠 수 있습니다. 예를 들어 "Spring API를 만들고 Docker로 배포"하면 Spring/Java + DevOps 템플릿이 모두 참조됩니다.

### 교정 이벤트 반영

대화 중 사용자가 방향을 수정한 부분은 생성되는 스킬의 **Rules** 섹션에 자동 반영됩니다.

예:
```
> 아니, JdbcCursorItemReader 말고 JpaPagingItemReader를 써줘.
```
→ 스킬 Rules에 "Reader는 JpaPagingItemReader를 사용" 추가

### 저장 위치 선택

- **프로젝트별** (`.claude/skills/<name>/SKILL.md`): 이 프로젝트에서만 사용하는 스킬
- **개인 글로벌** (`~/.claude/skills/<name>/SKILL.md`): 모든 프로젝트에서 공유하는 스킬

### 생성된 스킬 편집

저장된 SKILL.md는 일반 마크다운 파일이므로 직접 편집할 수 있습니다. 인터뷰에서 놓친 세부사항을 수동으로 추가하거나, 단계를 재배치할 수 있습니다.

## FAQ

### Q: 개발 프로젝트가 아닌 작업에도 사용할 수 있나요?

네. API 호출 자동화, 문서 분석/정리, 반복적인 시스템 관리 작업 등 **대화로 수행할 수 있는 모든 반복 작업**에 사용 가능합니다.

### Q: git 저장소가 없어도 되나요?

네. git 정보는 보조 컨텍스트로만 사용됩니다. 대화 기록만으로 스킬을 생성할 수 있습니다.

### Q: 생성된 스킬이 Skillify에 의존하나요?

아니요. 생성된 스킬은 **완전히 독립적**입니다. 도메인 템플릿은 스킬 생성 시에만 참조되고, 생성된 스킬 실행 시에는 사용되지 않습니다.

### Q: 다른 AI 에이전트 플랫폼에서도 사용할 수 있나요?

현재 버전은 **Claude Code 전용**입니다. 생성되는 SKILL.md 파일 자체는 agentskills.io 표준을 따르므로, 다른 플랫폼에서도 읽을 수 있지만 Skillify 자체는 Claude Code의 도구와 스킬 시스템에 의존합니다.

### Q: 인터뷰를 건너뛸 수 있나요?

현재 버전에서는 4라운드 인터뷰가 기본입니다. 단순한 작업의 경우 각 라운드의 질문 수가 적어 빠르게 진행됩니다.
